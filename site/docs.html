PhD OS Documentation

A personal research operating system built on Obsidian, rendered as a live GitHub Pages dashboard. Everything lives in markdown — the site is just a view over your vault.

Overview

PhD OS is a daily-driver dashboard for managing your PhD: tasks, papers, literature, books, calendar, and recovery — all in one place. The architecture has three layers:

  1. Obsidian vault — your markdown source of truth. You write notes; the system reads them.
  2. GitHub Actions build — runs on every push (and on a daily schedule). A Python script reads the vault and writes JSON files.
  3. Static site — vanilla HTML/CSS/JS served via GitHub Pages. Pages fetch the JSON files and render them. Live data (Todoist, Google Calendar) is fetched client-side at runtime.

Nothing requires a server, database, or login. All secrets (Todoist token, Google Calendar token) live only in your browser's localStorage.

Deployment

The site deploys automatically via GitHub Actions whenever you push to main. Obsidian Git auto-pushes every 10 minutes. The workflow also runs on a daily schedule so the Discover feed stays fresh.

Manual trigger

GitHub → your repo → Actions → Build PhD OS Dashboard → Run workflow. Use this after adding new books or papers when you don't want to wait for the auto-push.

Checking build logs

Actions → most recent run → click the job name. The Python script prints one line per JSON file written — confirm your new content was picked up. If the build failed, the error is here (usually a malformed YAML frontmatter colon in one of your notes).

Local build (for debugging)

Run from the vault root:

python3 .github/workflows/build_dashboard.py

This writes _site/ locally so you can inspect the generated JSON without waiting for a GitHub build.

Light / Dark Mode

PhD OS respects your OS color scheme by default. To force light mode regardless of OS preference, click the ☀ button in the top nav bar. Click the 🌙 that replaces it to return to system default. The preference is saved in localStorage under phd_theme and persists across all pages and sessions.

The early-load script theme.js (loaded synchronously in each page's <head>) applies the theme class before first paint, preventing flash.

Papers

A pipeline view of all your active research papers, grouped by stream and sorted by stage. Each paper is a markdown file in 100 Research/My Papers/. A progress bar shows position in the pipeline from ideation to published. Papers with status: active are shown; paused or abandoned are hidden.

The Projects page also shows a stream summary (paper count + stage pills per stream) and upcoming research tasks from Todoist filtered to the Research project and its sub-projects.

Discover

A daily-refreshed feed of IS journal papers fetched from OpenAlex at build time. Four sections: recent issues (last 6 months, newest first), trending this year (most-cited), AI Companions keyword search, and JudgyAI keyword search.

Click → Library on any card to jump to the Library page with the paper pre-filled. Click ✕ to hide a paper for the browser session. To permanently hide it, add its DOI to _dismissed.md in Source Papers and push — it is filtered at the next build.

The Article of the Day on the Home dashboard draws from the IS journals section of discover.json — it picks dayIndex % papers.length, so a new paper appears each day as long as the pool has papers.

Library

Two sections: Quick Add at the top, and your full literature library below with stream/keyword/related-paper filters.

Quick Add (DOI lookup)

Paste a DOI (or doi.org URL, ACM link, etc.) → click Look up → CrossRef returns title, authors, journal, year → OpenAlex is queried non-blocking for keywords → you adjust the fields → click Copy frontmatter. Paste the YAML into a new note in 100 Research/Source Papers/ and push.

You can also pre-fill the form from the Discover page by clicking → Library on any discovery card.

Books

A visual bookshelf showing all books in 400 Personal/Books/. Three sections: Currently Reading, Finished (sorted newest first), and Want to Read. Hover any cover for a popup with title, author, dates, and your snippet.

Cover images

Covers are fetched client-side at runtime — not at build time. When the page loads, each book spine without a manually-set cover_url triggers an asynchronous lookup:

  1. If isbn is set in the note: https://covers.openlibrary.org/b/isbn/{isbn}-L.jpg directly.
  2. If no ISBN: Open Library search API with title + author and sort=new (newest editions first). The first result with a cover_i is used — that cover ID reliably maps to a current-edition cover.
  3. Covers fade in as they load. If all sources fail, a colored placeholder with the book's first letter is shown.

Cover results are cached in _coverCache (in-memory, per page load) so navigating away and back doesn't re-fetch.

Quick Add (edition picker)

The Quick Add form at the top of the Books page lets you look up any book and generate Obsidian frontmatter with the correct cover URL, ISBN, and metadata.

  1. Type a title (and optionally author) → click Search.
  2. Open Library is queried for up to 3 matching works. Their editions are fetched in parallel and merged into one list sorted newest-first.
  3. A grid of edition cover thumbnails appears. Each card shows the edition-specific cover, year, publisher, and ISBN-13. Pick the edition whose cover matches your copy.
  4. A preview panel appears with the full cover and fields to fill in: title, author (editable), status, star rating, and tags.
  5. Click Copy frontmatter → paste into a new note in 400 Personal/Books/ → push.

The cover_url in the copied YAML points to that specific edition's cover via Open Library's cover ID. This bypasses the search-based lookup on future page loads — the URL is stored directly.

Recharge

A break timer backed by your activity list in 500 Recharge/Recharge List.md. Headings in that file become category labels; bullet points become activities.

How to use it

  1. Browse the activity list and click Start next to what you want to do.
  2. A timer starts counting up and appears prominently at the top of the page. The timer persists across page navigations — startedAt is stored in localStorage, so the elapsed time is always computed from the saved start time, not a running interval.
  3. When done, click Stop & Log. The session (activity, category, start/end time, duration) is saved to your session log in localStorage.
  4. Cancel discards the session without logging it.

Today's sessions are shown at the bottom of the page with total minutes. The Home dashboard Recharge card shows the active timer (if running) or time since your last break, linking to this page.

To add or change activities: edit 500 Recharge/Recharge List.md in Obsidian (use markdown headings for categories, bullet points for items) and push. The page reads recharge.json which is rebuilt on each push.

Todoist Integration

The website fetches live task and project data from Todoist's REST API (https://api.todoist.com/api/v1). Your API token is stored in localStorage under phd_todoist_token and is never sent anywhere except directly to Todoist over HTTPS.

Finding your token

todoist.com → Settings → Integrations → Developer → API token. Copy it and paste it into the modal that appears on first visit (or after clearing it).

Clearing / re-entering

Click the ⚙ gear icon in the top nav → confirm → the token is cleared and you'll be prompted on the next page load.

Google Calendar Integration

The Home dashboard "Today's Meetings" card fetches events from all your Google Calendars using the Google Calendar REST API with OAuth 2.0 implicit flow.

One-time setup

  1. Go to console.cloud.google.com.
  2. Create a new project → enable the Google Calendar API.
  3. Credentials → Create Credentials → OAuth Client ID → Web application.
  4. Under Authorized JavaScript origins, add your GitHub Pages URL (e.g. https://marvelmarauder.github.io). You can also add http://localhost for local testing.
  5. Copy the Client ID — it looks like 123456789-abc.apps.googleusercontent.com.
  6. In site/app.js, find const GCAL_CLIENT_ID = '...' near the bottom and paste your Client ID.
  7. Push to main. On next load, click Connect Google Calendar.
The Client ID is not a secret — it is safe to commit to a public repo. It only works from the authorized origin you registered.

Token persistence and silent refresh

After you connect, the OAuth access token is stored in localStorage as phd_gcal_auth (access token + expiry). Tokens expire after 1 hour. On each page load, if a previous connection exists (phd_gcal_connected = 1 in localStorage), the site attempts a silent token refresh via requestAccessToken({prompt:''}) before showing a reconnect button. If you are still signed into Google in your browser, this is invisible. The reconnect button only appears if your Google session itself has ended.

What it reads

All calendars in your Google account with selected !== false (i.e. everything checked in your calendar list). Events for today only (timeMin = start of today, timeMax = start of tomorrow). All-day events show "All day"; timed events show the start time. Meet/Zoom badges are detected from hangoutLink or the event's location URL.

Email Pipeline — Overview

The email pipeline reads emails from a designated Apple Mail folder, runs them through a local LLM, accumulates proposed tasks in a queue, and lets you review and approve them through a local web UI. Everything runs on your Mac — emails never leave it. The only data that goes anywhere is the task text you explicitly approve, which goes to Todoist.

How it works

  1. You create a folder called LLM Queue in Apple Mail and set up Outlook rules to route emails there.
  2. A LaunchAgent runs email_pipeline.py every 2 hours. Mail opens automatically if closed.
  3. Emails are parsed by the phd-email-parser Ollama model. Proposed tasks, a 600-char snippet, and any PDF attachments are written to ~/.phd_os_queue.json and ~/.phd_os_attachments/.
  4. Emails are moved to LLM Processed immediately after parsing.
  5. Open PhD OS Review.app from your Dock. Select project + due date for each task, read the email snippet, download PDFs, then click Add to Todoist.

Email Pipeline — Setup

Install Ollama first. Download at ollama.com.

Step 1 — Pull the base model

ollama pull llama3.2:3b

Step 2 — Run setup

bash scripts/setup.sh

Creates ~/.phd_os_config.json, builds the custom model, installs the LaunchAgent.

Step 3 — Add Exchange account to Apple Mail

System Settings → Internet Accounts → Add Account → Microsoft Exchange. You don't need to use Mail as your primary client — just add the account so it syncs in the background. IMAP and the new Outlook's API are both blocked by university IT; Apple Mail + AppleScript is the only path.

Step 4 — Create folders and rules in Outlook

Create two folders: LLM Queue and LLM Processed. They sync to Apple Mail. Set up Outlook rules to route emails into LLM Queue. Suggested tiers:

  • Senders: advisor, committee, department admin → LLM Queue
  • Keywords: "deadline", "action required", "review", "submit" → LLM Queue
  • Journal alerts: misq.org, aisnet.org, etc. with "new issue" → LLM Queue

Config reference

~/.phd_os_config.json:

{
  "todoist_token": "your-todoist-api-token",
  "ollama_model": "phd-email-parser",
  "ollama_url": "http://localhost:11434/api/generate",
  "outlook_queue_folder": "LLM Queue",
  "outlook_processed_folder": "LLM Processed",
  "max_emails_per_run": 20
}

Email Pipeline — Daily Use

The pipeline runs automatically every 2 hours. When ready to review, open PhD OS Review.app from your Dock — it opens http://localhost:7891. If the server is already running, it goes straight to the browser.

# Rebuild the Dock app (e.g. if vault path changed)
bash scripts/build_app.sh

# Run the server manually
python3 scripts/review_server.py

# CLI fallback (no web UI needed)
python3 scripts/review_tasks.py

# Run pipeline manually right now
python3 scripts/email_pipeline.py

# Process but leave emails in LLM Queue
python3 scripts/email_pipeline.py --no-move

# Parse and print — write nothing, move nothing
python3 scripts/email_pipeline.py --dry-run

# View the last run log
cat /tmp/phd_os_pipeline.log

# Uninstall the LaunchAgent
launchctl unload ~/Library/LaunchAgents/com.phd-os.email-pipeline.plist

Email Pipeline — Review UI

Local web app served by review_server.py on port 7891. Three tabs.

Task Queue tab

Each batch shows one card per email. The card header shows sender + subject. Below that:

  • Summary — LLM one-sentence summary of required action.
  • Show email — collapsible showing first 600 chars of the body.
  • 📎 links — PDF attachment downloads (stored in ~/.phd_os_attachments/).

Each proposed task row has: checkbox (checked by default), editable task text, priority badge, project dropdown (populated from Todoist API — this is the most important field to review), due date input, and × dismiss button.

Dismissing a task shows a reason field. The reason is appended to scripts/model/corrections.md and baked into the model on the next rebuild — this is how the model improves over time.

Click Add to Todoist to create all checked tasks. Run Pipeline Now triggers email_pipeline.py immediately and streams its log. Stop Server shuts down the server gracefully.

Add Example tab

Paste a real email, write the correct output (summary + tasks), save to scripts/model/examples.md. Use this when the LLM handled an email badly. Rebuild the model after adding examples.

Model tab

Rebuilds phd-email-parser from the Modelfile, examples, and corrections. Takes 30–60 seconds. Rebuild log streams to the page.

Fine-Tuning the Model

The model is a customized llama3.2:3b with your context baked in via two files in scripts/model/:

  • Modelfile — system prompt, temperature, base model declaration
  • examples.md — email→task pairs (few-shot examples)
  • corrections.md — rejection reasons from the review UI (auto-populated)

Data Flow

Website pipeline

Entirely build-time for vault content. Runtime for live data (Todoist, GCal, book covers).

  1. You edit a markdown note in Obsidian (e.g. add a source paper).
  2. Obsidian Git pushes to main on GitHub (auto, every 10 min).
  3. GitHub Actions runs build_dashboard.py:
    • Glob-scans vault folders, reads YAML frontmatter from each note.
    • Calls OpenAlex API for IS journal discovery papers and stream searches → discover.json.
    • Writes papers.json, lit.json, books.json, recharge.json, discover.json into _site/.
    • Copies all files from site/ → _site/.
    • Note: build_books() does NOT fetch cover images from any API. It only passes through isbn and any manually-set cover_url from the note frontmatter. Covers are loaded client-side.
  4. GitHub Pages deploys _site/.
  5. Browser renders a page:
    • theme.js runs synchronously — applies theme-light class if stored preference exists.
    • app.js and style.css load.
    • Page-specific JS fetches the relevant JSON file (fetch('books.json'), etc.) and renders it.
    • Todoist: GET /api/v1/tasks called live with token from localStorage.
    • Google Calendar: If phd_gcal_connected is set, silent token refresh is attempted via GIS. If successful, events are fetched from the Calendar API. If the Google session has ended, a reconnect button appears.
    • Book covers: Each .book-spine without data-cover-loaded triggers fetchBookCover(isbn, title, author) — searches Open Library with sort=new, gets cover_i, constructs the cover URL. Results are cached in _coverCache for the session.
    • Article of Day: Reads discover.json journals section, picks papers[dayIndex % papers.length].
    • LDS Quote: Reads hardcoded 50-quote array in app.js, picks LDS_QUOTES[dayIndex % 50]. No network call.

Email pipeline

Entirely local. Nothing leaves your Mac except task text approved to Todoist.

  1. Trigger: LaunchAgent fires every 2 hrs, or you run manually / click Run Pipeline Now in the UI.
  2. Mail open: If Mail isn't running, script opens it, polls up to 60s, then waits 15s for Exchange to sync.
  3. AppleScript fetch: Script generates and runs an AppleScript via osascript (Popen-based polling loop — avoids the macOS hang where communicate() after kill() blocks indefinitely). The AppleScript iterates all mailboxes of each Exchange account to find "LLM Queue" (direct mailbox lookup fails on Exchange with error -1728 — iteration is the fix), synchronizes it, then reads each message: writes body to a private temp dir (mode 700), saves PDF attachments, returns a tab-delimited line per message (id, subject, sender, date, pdf names).
  4. Deduplication: Emails sharing a normalized thread subject (Re:/Fwd: stripped) keep only the most recent message.
  5. Ollama parse: Each email sent to http://localhost:11434/api/generate with model phd-email-parser. Parses the JSON response for summary and todos.
  6. Queue write: Tasks, 600-char snippet, and attachment filenames atomically appended to ~/.phd_os_queue.json (written to .tmp file first, then os.replace()). PDFs copied from temp to ~/.phd_os_attachments/ before cleanup.
  7. Move: Second AppleScript moves all processed messages to "LLM Processed".
  8. Review: review_server.py reads queue.json and renders task cards. Fetches Todoist project list for the dropdown.
  9. Approve: POST https://api.todoist.com/api/v1/tasks with project_id and due_string. Approved + dismissed tasks removed from queue.json.

Build Script

Located at .github/workflows/build_dashboard.py. Runs from the repo root. Key functions:

build_papers()100 Research/My Papers/*.mdpapers.jsonSorts by stream, then by stage index

build_books()400 Personal/Books/*.mdbooks.jsonDoes NOT fetch covers. Passes through isbn and any manually-set cover_url only. Cover images are loaded client-side at runtime.

The script copies all files from site/ to _site/, then writes the JSON files. GitHub Pages serves _site/ as the root.

Security & Protections

PhD OS runs with elevated trust on your Mac. The following protections limit what it can actually do.

macOS automation permission scope

The pipeline reads only from the LLM Queue folder and moves only to LLM Processed. It never sends email, deletes email, reads Inbox, or accesses Contacts or Calendar. The macOS permission dialog is intentionally broad — per-folder scoping is not offered by Apple. The limitations above are enforced by the script, not the OS.

AppleScript injection prevention

Folder names from ~/.phd_os_config.json are embedded into AppleScript strings. A strict regex allows only letters, digits, spaces, hyphens, and underscores — any other character exits before any AppleScript runs. Common system folder names (Inbox, Sent, Drafts, Trash) are also blocked.

Atomic queue writes

~/.phd_os_queue.json is always written to a .tmp file first, then swapped in via os.replace(). A crash mid-write leaves the previous queue intact.

Private temp directory

Email bodies and PDF attachments are written to a temp dir with mode 700 (owner read/write only), deleted in a finally block even if the script crashes. PDFs are copied to ~/.phd_os_attachments/ before temp cleanup.

Config file permissions

~/.phd_os_config.json is created with mode 600 by setup.sh. Verify: ls -l ~/.phd_os_config.json should show -rw-------. Fix: chmod 600 ~/.phd_os_config.json.

Cover image URL validation

Cover URLs from books.json are validated before use in <img src> — only http:// and https:// schemes are accepted. javascript: and data: URLs are silently replaced with a placeholder. All other user-controlled strings go through escHtml().

Todoist token storage

Stored in localStorage for the site's origin only. Sent only to api.todoist.com over HTTPS. No third-party JS on any page. Visible in DevTools — treat it like a password.

Human approval gate

The pipeline never creates Todoist tasks automatically. Every proposed task goes through the review UI (or CLI) before any API call.

Email cap

Default: 20 emails per run. Override in config: "max_emails_per_run": 10. Prevents runaway processing from flooded queue folders.

Troubleshooting

Site shows stale data after push

Check the Actions tab on GitHub. If the build failed, click the run — the error is usually a malformed YAML frontmatter block (unquoted colon in a title field). Fix the note and push again.

Book covers not loading or showing wrong cover

Covers are fetched client-side from Open Library at page load time. If covers aren't appearing: open DevTools → Network tab → look for failed requests to openlibrary.org. If Open Library is unreachable or rate-limiting, covers will fail silently and show placeholders — try again later.

If the cover is showing the wrong edition: use the Quick Add form on the Books page to find your edition and copy its frontmatter — the cover_url in the generated YAML pins the exact cover and bypasses the search entirely.

If you know the exact cover you want: find it on openlibrary.org, get the cover ID from the URL, and set cover_url: https://covers.openlibrary.org/b/id/{id}-L.jpg directly in the book's frontmatter.

Light mode not persisting between pages

Check localStorage.getItem('phd_theme') in the browser console — it should return 'light' after you click ☀. If it does but pages still go dark, check that theme.js is present in _site/ (it should be — the build script copies all of site/). The class html.theme-light must be applied before CSS renders; theme.js runs synchronously in <head> for this reason.

Google Calendar keeps asking to reconnect

The OAuth token expires every hour. Silent refresh should handle this automatically if you're still signed into Google in the browser. If you keep seeing the reconnect button: check localStorage.getItem('phd_gcal_connected') — should be '1'. If it's null, the silent refresh can't try. Connect once manually and it will be set.

Article of the Day not updating

The article rotates daily using dayIndex % papers.length from the IS journals section of discover.json. If the same article appears for many days, the pool is small — trigger a manual GitHub Actions build to refresh discover.json with new OpenAlex results.

Recharge timer lost after closing browser

The active timer is stored in localStorage.phd_recharge_active as {item, category, startedAt}. Closing the browser preserves this — it will resume on next visit. If it's gone, check DevTools → Application → Local Storage. The timer was likely stopped or the localStorage was cleared.

Discover feed empty

The OpenAlex fetch happens at build time. If a build ran while OpenAlex was unreachable, sections may be empty. Trigger a manual build to refresh.

Todoist token reset after switching browsers

The token lives in localStorage scoped to the specific browser and profile. Private/incognito windows don't share it. Re-enter it in the new browser; it takes about 5 seconds.

Email pipeline: 0 tasks proposed every run

Check /tmp/phd_os_pipeline.log for the specific message. Most common: "LLM Queue is empty" (Outlook rules aren't routing emails there, or Mail hasn't synced yet). Also check the folder name in ~/.phd_os_config.json matches exactly — it's case-sensitive.

Email pipeline: Ollama errors

ollama serve in terminal, or check the Ollama menu bar app. Then ollama list — phd-email-parser must appear. If missing: bash scripts/model/build_model.sh.

Email pipeline: AppleScript permission errors

System Settings → Privacy & Security → Automation → make sure Terminal (or your Python launcher) has permission to control Mail. A one-time dialog may appear the first time — click Allow.

Review UI: project dropdown empty

The dropdown fetches from Todoist on page load. Check ~/.phd_os_config.json has a valid todoist_token. Test: curl -s https://api.todoist.com/api/v1/projects -H 'Authorization: Bearer YOUR_TOKEN'. If that returns data, restart the review server.

Church page: Zoom buttons say "Not set"

Zoom links live in Credentials (localStorage.phd_church_credentials). Click Edit on the Church page to fill them in. They never persist to git — re-enter them if you clear localStorage or use a different browser.
