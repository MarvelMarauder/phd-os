PhD OS Documentation

A personal research operating system built on Obsidian, rendered as a live GitHub Pages site. Everything lives in markdown — the site is just a view over your vault.

Overview

PhD OS is a daily-driver dashboard for managing your PhD: papers, literature, tasks, books, and recovery — all in one place. The architecture has three layers:

  1. Obsidian vault — your markdown source of truth. You write notes; the system reads them.
  2. GitHub Actions build — runs daily (and on every push). A Python script reads the vault and writes JSON files.
  3. Static site — vanilla HTML/CSS/JS served via GitHub Pages. Pages fetch the JSON files and render them.

Nothing requires a server, database, or login. The Todoist integration is purely client-side (your API token lives in localStorage).

Deployment

The site deploys automatically via GitHub Actions whenever you push to main. The workflow also runs on a daily schedule so the discovery feed stays fresh even without a push.

Manual trigger

Go to your repo on GitHub → Actions → Build PhD OS Dashboard → Run workflow. Useful after adding new books or papers when you don't want to wait for the daily run.

Checking build logs

Actions → the most recent run → click the job name. The Python script prints a line for each JSON file it writes, so you can confirm your new content was picked up.

Home Dashboard

Live task view pulled directly from Todoist. Shows tasks due today, tasks due this week, and all active tasks grouped by project. Requires your Todoist API token (stored in your browser — never sent anywhere except Todoist).

Find your token at todoist.com → Settings → Integrations → Developer. Click the settings gear (⚙) in the top nav to clear and re-enter it.

Papers

A pipeline view of all your research papers, grouped by research stream and sorted by stage. Each paper is a markdown file in 100 Research/My Papers/. The Kanban-style columns show the stage of each paper from ideation to published.

Discover

A daily-refreshed feed of new papers from the 11 top IS journals, fetched from OpenAlex at build time. Three sections: recent issues (last 6 months), trending this year (most-cited), and per-stream keyword searches.

Click → Library on any card to jump to the Library page with the paper's metadata pre-filled. Click ✕ to hide a paper for the session. To permanently hide it, add its DOI to _dismissed.md and push — it will be filtered out at the next build.

Library

Two parts: a Quick Add form at the top (paste a DOI → fetches metadata from CrossRef → generates a YAML frontmatter block you can paste into a new note), and your full literature library below it, filterable by stream and searchable by keyword.

Books

Visual bookshelf grid showing all books in 400 Personal/Books/. Hover any cover to see details. Three sections: Currently Reading, Finished (sorted newest first), and Want to Read.

Recharge

A curated list of activities to decompress. Reads from 500 Recharge/Recharge List.md — headings become section titles, bullet points become items. Useful for ADHD-friendly context switching.

Email Pipeline — Overview

The email pipeline reads emails from a designated Apple Mail folder, runs them through a local LLM (Ollama), and accumulates proposed to-do items in a queue file. You review and approve the queue through a local web UI whenever you want — the pipeline runs automatically every two hours in the background.

All processing is local. Emails never leave your Mac. The only data that leaves is the task text you explicitly approve, which goes to Todoist. See Security & Protections for a full breakdown of what the pipeline can and cannot do.

How it works

  1. Create a folder called LLM Queue in Apple Mail. Set up rules in Outlook to forward emails you want processed into it — they sync to Apple Mail automatically.
  2. A LaunchAgent runs email_pipeline.py every 2 hours. If Apple Mail isn't open, it launches it and waits for Exchange to sync before reading.
  3. Each email is parsed by the phd-email-parser Ollama model. Proposed tasks, an email snippet, and any PDF attachments are saved to ~/.phd_os_queue.json and ~/.phd_os_attachments/.
  4. Emails are moved to LLM Processed immediately after parsing (so the queue folder stays clean).
  5. Open PhD OS Review.app from your Dock to launch the review web UI. Select a project and due date for each task, read the email snippet if you need context, download any PDF attachments, then click Add to Todoist.

Email Pipeline — Setup

Before running setup, make sure Ollama is installed. Download it at ollama.com.

Step 1 — Pull the base model

In terminal:

ollama pull llama3.2:3b

This downloads a ~2GB model that fits comfortably in 16GB RAM and runs fast on Apple Silicon.

Step 2 — Run the setup script

bash scripts/setup.sh

The script will:

  • Create ~/.phd_os_config.json with your Todoist token
  • Build the custom phd-email-parser Ollama model
  • Install and load the LaunchAgent (runs every 2 hours)

Step 3 — Add your university account to Apple Mail

The pipeline reads emails via Apple Mail's AppleScript API (the new Outlook for Mac removed AppleScript support). You don't need to use Apple Mail as your main client — just add the account so it syncs in the background.

Go to System Settings → Internet Accounts → Add Account → Microsoft Exchange and sign in with your university credentials. Once connected, your Outlook folders (including LLM Queue) will be visible in Apple Mail.

Step 4 — Set up folders and rules in Outlook

Create two folders in Outlook: LLM Queue and LLM Processed. They will automatically sync to Apple Mail. Then set up rules in Outlook to move emails you want parsed into LLM Queue. Suggested rule tiers:

  • Tier 1 (senders): advisor, committee members, department admin → always route to LLM Queue
  • Tier 2 (keywords): subject contains "deadline", "action required", "review", "submit" → route to LLM Queue
  • Tier 3 (journal alerts): from misq.org, aisnet.org, springer.com with "new issue" or "table of contents" → route to LLM Queue

Config file reference

Located at ~/.phd_os_config.json:

{
  "todoist_token": "your-todoist-api-token",
  "ollama_model": "phd-email-parser",
  "ollama_url": "http://localhost:11434/api/generate",
  "outlook_queue_folder": "LLM Queue",
  "outlook_processed_folder": "LLM Processed"
}

Email Pipeline — Daily Use

You don't need to do anything for the pipeline to run — it fires automatically every 2 hours when your Mac is on. Apple Mail opens itself if it's closed.

When you're ready to review accumulated tasks, click PhD OS Review.app in your Dock. It opens a local web UI at http://localhost:7891 in your browser. If the server is already running, it goes straight to the browser.

To rebuild the app (e.g. if the vault path changed):

bash scripts/build_app.sh

If you prefer the terminal:

python3 scripts/review_server.py

There is also a minimal CLI fallback if the web UI isn't available:

python3 scripts/review_tasks.py

Useful terminal commands

# Check the pipeline log
cat /tmp/phd_os_pipeline.log

# Run the pipeline manually right now
python3 scripts/email_pipeline.py

# Run without moving emails (process but leave in LLM Queue)
python3 scripts/email_pipeline.py --no-move

# Parse emails and print proposed tasks — write nothing, move nothing
python3 scripts/email_pipeline.py --dry-run

# See how many tasks are pending
python3 -c "import json; q=json.load(open('$HOME/.phd_os_queue.json')); print(sum(len(e['todos']) for b in q['batches'] for e in b['emails']), 'pending')"

# Uninstall the LaunchAgent
launchctl unload ~/Library/LaunchAgents/com.phd-os.email-pipeline.plist

Email Pipeline — Review UI

The review UI is a local web app served by review_server.py on port 7891. It has three tabs.

Task Queue tab

This is the main working view. Each batch shows the emails processed in that run, grouped as cards. For each email you see:

  • From / Subject — sender name and subject line in the card header.
  • Summary — one-sentence LLM summary of what the email is about and what action it requires, shown in italics below the header.
  • Show email — a collapsible that reveals the first ~600 characters of the email body so you can read the actual context without switching to Mail.
  • 📎 attachment links — if the email had PDF attachments, download links appear here. Attachments are stored in ~/.phd_os_attachments/ and survive between sessions.

Each proposed task appears below its email card as a row with:

  • Checkbox — checked by default. Uncheck to exclude a task from the current approval batch without dismissing it.
  • Editable task text — click to edit before approving. Changes are sent to Todoist as-is.
  • Priority badge — Urgent / High / Normal / Low, set by the LLM.
  • Project dropdown — your full Todoist project list, loaded from the API on page open. The LLM's project_hint pre-selects the closest match; change it before approving. This is the most important field to review — a task sent without a project lands in Todoist's inbox.
  • Due date input — pre-filled with the LLM's suggestion. Accepts natural language (today, this week, Friday, 2026-06-15). Clear it to create the task with no due date.
  • × dismiss button — removes the task. A reason field appears; your reason is saved to scripts/model/corrections.md and gets baked into the model on the next rebuild, creating a feedback loop.

Click Add to Todoist in the bottom bar to create all checked tasks at once. The bar shows how many tasks are selected. Tasks are removed from the queue immediately after creation.

The Run Pipeline Now button at the top triggers email_pipeline.py in the background and streams its log output below the button. Useful if you want to process new emails without waiting for the next 2-hour automatic run.

Add Example tab

Paste in a real email, write the correct summary and tasks, and save it to scripts/model/examples.md. These become few-shot examples baked into the model. Use this when you encounter an email the LLM handled badly. After adding examples, go to the Model tab and rebuild.

Model tab

Rebuilds the phd-email-parser Ollama model from scripts/model/Modelfile, examples.md, and corrections.md. Takes about 30–60 seconds. The rebuild log streams to the page so you can see when it finishes.

Click Stop Server in the header to shut down the web server gracefully when you're done. You don't need to keep it running — the pipeline runs independently of the review server.

Fine-Tuning the Model

The phd-email-parser model is a customized version of llama3.2:3b with your research context and task categories baked in. You can sharpen it over time by adding examples — no GPU or training required.

How it works

The model is defined by two files in scripts/model/:

  • Modelfile — sets the system prompt, temperature, and baked-in context about you and your work
  • examples.md — email→task pairs you curate. These become few-shot examples embedded in the model.

Adding examples

Open scripts/model/examples.md and add a new block following this format:

## Example N — brief description

INPUT:
FROM: sender@example.com
SUBJECT: Email subject here
BODY:
The email body text goes here.

OUTPUT:
{
  "summary": "One sentence summary.",
  "todos": [
    {
      "task": "Concrete action item",
      "priority": 2,
      "project_hint": "research",
      "due_suggestion": "this week"
    }
  ]
}

After adding examples, rebuild the model:

bash scripts/model/build_model.sh
The best examples come from real emails the model got wrong. When you review a task and have to heavily edit it, that's a sign the original email would make a good training example.

Editing the system prompt

Open scripts/model/Modelfile directly to adjust the system prompt — for example, to add new task categories, update your research stream descriptions, or add recurring sender context (e.g. "emails from church@example.org are always church admin"). Rebuild after any changes.

Security & Protections

PhD OS runs with elevated trust on your Mac — it can control Apple Mail via AppleScript and write to Todoist. The following protections limit what it can actually do.

macOS automation permission scope

When macOS asks whether Terminal/Python can control Mail, that permission is broad — it technically allows any script to read or send any email. The email pipeline limits itself well below that ceiling:

  • Reads only from the folder named in outlook_queue_folder (default: LLM Queue)
  • Moves only to the folder named in outlook_processed_folder (default: LLM Processed)
  • Never sends email, never deletes email, never reads Inbox or any other folder
  • Never accesses Contacts, Calendar, or any other app
The permission macOS shows you is intentionally broad — Apple does not offer per-folder granularity. The limitations above are enforced by the script, not by the OS.

AppleScript injection prevention

Folder names from ~/.phd_os_config.json are embedded into AppleScript strings at runtime. A crafted folder name could escape the string and execute arbitrary AppleScript.

The pipeline prevents this with a strict regex before touching anything: folder names may only contain letters, digits, spaces, hyphens, and underscores. Any other character causes the script to exit with an error before any AppleScript runs. Common system folder names (Inbox, Sent, Drafts, Trash, etc.) are also blocked.

Dry-run and no-move modes

You can always verify what the pipeline would do before it does anything:

# Read and parse emails, print proposed tasks — write nothing, move nothing
python3 scripts/email_pipeline.py --dry-run

# Process and queue tasks, but leave emails in LLM Queue
python3 scripts/email_pipeline.py --no-move

Human approval before any Todoist write

The pipeline never creates Todoist tasks automatically. Every proposed task goes through review_tasks.py, which shows you each task individually (approve / edit / skip), then prints a final summary of everything about to be created and asks for confirmation before making a single API call.

Private temp directory

Email bodies and PDF attachments are written to a private temp directory created at the start of each pipeline run with mode 700 (readable only by you). The directory and all its files are deleted in a finally block that runs even if the script crashes. Other users on the same machine cannot read your email content while the pipeline is running.

Config file permissions

~/.phd_os_config.json contains your Todoist API token. setup.sh creates it with mode 600 (owner read/write only). If you create or edit the file manually, verify the permissions with:

ls -l ~/.phd_os_config.json
# Should show: -rw------- (600)

If it shows anything else: chmod 600 ~/.phd_os_config.json

Token not echoed during setup

setup.sh uses read -s when prompting for the Todoist token, so it is never displayed on screen. The token is passed to Python via an environment variable (not shell interpolation), so it is JSON-encoded correctly even if it contains special characters.

Atomic queue file writes

The pending task queue (~/.phd_os_queue.json) is written atomically: the script writes to a .tmp file first, then does an os.replace() to swap it in. A crash mid-write leaves the old queue intact rather than producing a corrupted file. The queue file is also created with mode 600.

Email cap

The pipeline processes at most max_emails_per_run emails per run (default: 20). A flooded LLM Queue folder — from a runaway forwarding rule, for example — cannot cause unbounded API calls or runaway Ollama usage. Override in ~/.phd_os_config.json:

"max_emails_per_run": 10

Audit log

Every action taken by the pipeline is written as a JSON line to /tmp/phd_os_audit.log: which emails were read, when they were moved, how many tasks were queued. You can inspect it at any time:

cat /tmp/phd_os_audit.log | python3 -m json.tool --no-indent | head -40
The audit log lives in /tmp and is cleared on reboot. It is not a tamper-proof security record — it is a convenience log for your own review.

Cover image URL scheme validation

Book cover URLs from books.json are validated before being placed in an <img src> attribute. Only http:// and https:// URLs are accepted; anything else (including javascript: or data: URLs) is silently replaced with a colored placeholder. All other user-controlled strings rendered into HTML go through escHtml().

Todoist token storage

The Todoist API token used by the website is stored in your browser's localStorage for the site's origin. It is never sent anywhere except directly to the Todoist API over HTTPS. There is no third-party JavaScript on any page, so there is no script that could exfiltrate it. It is visible in browser developer tools — treat it like a password and don't use a shared machine.

What is NOT protected

Being honest about limitations:

  • The macOS automation permission cannot be scoped to specific folders — a malicious script running as you could abuse it. Revoke it in System Settings → Privacy & Security → Automation if the pipeline is not in use.
  • The audit log in /tmp is not append-only or cryptographically signed — it can be modified or deleted.
  • The Todoist token in localStorage is visible to browser dev tools. On a shared machine, clear it with the ⚙ button in the nav when done.
  • The pipeline scripts have not been independently security-audited.

Data Flow

Website pipeline

This pipeline is purely build-time. Nothing runs on a server at request time.

  1. You write or edit a markdown note in Obsidian (e.g. a new source paper in 100 Research/Source Papers/).
  2. Obsidian Git pushes the change to main on GitHub (every 10 minutes).
  3. GitHub Actions detects the push and runs build_dashboard.py. The script glob-scans the vault folders, reads YAML frontmatter, makes API calls to OpenAlex/Open Library, and writes JSON files (papers.json, lit.json, books.json, recharge.json, discover.json) into _site/. It also copies everything from site/ into _site/.
  4. GitHub Pages deploys _site/ as a static site.
  5. When you open a page in your browser, the HTML loads app.js and style.css, then fetches the relevant JSON file (fetch('lit.json')), and renders the data.
  6. Live data (Todoist tasks, Google Calendar events) is fetched client-side at runtime using tokens stored in localStorage. This never touches the build.
If a page looks stale, the build is the bottleneck — not the browser cache. Trigger a manual GitHub Actions run to force a rebuild.

Email pipeline

This pipeline is entirely local. Each step runs as a separate process on your Mac.

  1. Trigger: The LaunchAgent fires email_pipeline.py every 2 hours (or you run it manually / click "Run Pipeline Now" in the web UI).
  2. Mail open: If Apple Mail isn't running, the script opens it and waits up to 60 seconds for it to launch, then adds a 15-second delay for Exchange to sync. This is why the pipeline can be slow if Mail was closed.
  3. AppleScript fetch: The script generates an AppleScript and runs it via osascript. The AppleScript iterates all mailboxes to find the one named "LLM Queue", calls synchronize on it, then reads each message: writes the body to a private temp file, saves any PDF attachments to temp, and returns a tab-delimited line per message (id, subject, sender, date, pdf names).
  4. Deduplication: If multiple messages in the queue share the same thread subject (e.g. a back-and-forth), only the most recent one is processed.
  5. Ollama parse: For each email, the script sends a prompt to http://localhost:11434/api/generate with the email content and asks for a JSON object with summary and todos. The model is phd-email-parser.
  6. Queue write: The resulting tasks, a 600-char body snippet, and a list of saved attachment filenames are atomically appended to ~/.phd_os_queue.json. PDFs are copied from temp to ~/.phd_os_attachments/ before cleanup.
  7. Move: A second AppleScript moves all messages from "LLM Queue" to "LLM Processed".
  8. Review: When you open the Review UI, review_server.py reads queue.json and renders the task cards. It also fetches your Todoist project list from the API to populate the project dropdown.
  9. Approve: Clicking "Add to Todoist" sends each checked task (with your selected project ID and due date) to POST https://api.todoist.com/api/v1/tasks. Approved and dismissed tasks are removed from queue.json.

Debugging by Hand

When something breaks, you can inspect every stage of each pipeline independently. You don't need Claude to read logs — all the state is in plain text files.

Debugging the email pipeline

# 1. Check the last pipeline run log
cat /tmp/phd_os_pipeline.log

# 2. Check the audit trail (JSON lines — one per action)
cat /tmp/phd_os_audit.log

# 3. See what's in the queue right now
python3 -c "
import json, sys
q = json.load(open('$HOME/.phd_os_queue.json'))
for b in q['batches']:
    print('Batch:', b['run_at'])
    for e in b['emails']:
        print(' ', e['from'], '|', e['subject'])
        for t in e['todos']:
            print('   -', t['task'], '|', t.get('project_hint'), '|', t.get('due_suggestion'))
"

# 4. Count pending tasks
python3 -c "import json; q=json.load(open('$HOME/.phd_os_queue.json')); print(sum(len(e['todos']) for b in q['batches'] for e in b['emails']), 'tasks pending')"

# 5. Run the pipeline manually with dry run (no writes, no moves)
python3 scripts/email_pipeline.py --dry-run

# 6. Run the pipeline and leave emails in place (process but don't move)
python3 scripts/email_pipeline.py --no-move

# 7. Check if Ollama is running and the model exists
curl -s http://localhost:11434/api/tags | python3 -m json.tool
ollama list

# 8. Test Ollama directly with a fake email
curl -s http://localhost:11434/api/generate \
  -d '{"model":"phd-email-parser","prompt":"FROM: test@test.com\nSUBJECT: Test\nBODY: Please review the attached paper by Friday.","stream":false}' \
  | python3 -c "import json,sys; print(json.load(sys.stdin)['response'])"

# 9. Check if the LaunchAgent is loaded and when it last ran
launchctl list | grep phd-os

# 10. Check Apple Mail is seeing the LLM Queue folder
osascript -e 'tell application "Mail" to get name of every mailbox of every account' 2>&1 | tr , '\n' | grep -i llm

# 2. Inspect a generated JSON file directly

Reading the queue file by hand

~/.phd_os_queue.json is plain JSON. Its structure:

{
  "batches": [
    {
      "run_at": "2026-05-21T14:32:00+00:00",
      "emails": [
        {
          "subject": "IRB revision needed",
          "from": "irb@university.edu",
          "date": "Wednesday, May 21, 2026 at 9:15:00 AM",
          "snippet": "First 600 chars of the email body...",
          "summary": "IRB requires protocol revision before approval.",
          "attachments": [
            {"name": "revision-notes.pdf", "file": "12345_revision-notes.pdf"}
          ],
          "todos": [
            {
              "task": "Submit IRB protocol revision",
              "priority": 1,
              "project_hint": "research",
              "due_suggestion": "this week"
            }
          ]
        }
      ]
    }
  ]
}

You can edit this file directly to fix corrupted entries, remove a stuck batch, or clear the whole queue:

# Clear entire queue (nuclear option)
echo '{"batches":[]}' > ~/.phd_os_queue.json

# Pretty-print the whole queue
python3 -m json.tool ~/.phd_os_queue.json
The queue file is written atomically via a temp file + rename — but if you edit it by hand while the review server is running, your changes may be overwritten when the server next saves the queue. Stop the server first (Stop Server button), edit, then restart.

Troubleshooting

Site isn't updating after I push

Check the Actions tab on GitHub. If the build failed, click the run to see the error. Common cause: a malformed YAML frontmatter block in one of your notes (e.g. an unquoted colon in a title).

Book cover isn't showing

The cover is fetched at build time, not runtime. If you added the book recently, trigger a manual build. If it still doesn't show, try adding the isbn field — the ISBN lookup is more reliable than the title search. As a last resort, find a cover image URL and set cover_url directly.

Discover feed shows no papers

The OpenAlex fetch happens at build time. If a build ran while OpenAlex was unreachable, discover.json may have empty sections. Trigger a manual build to refresh.

Todoist token keeps getting cleared

The token lives in localStorage for your browser profile. If you switch browsers, use a private window, or clear site data, you'll need to re-enter it. This is intentional — the token is never sent to any server other than Todoist.

Email pipeline: "No emails in queue"

Check that your Outlook rules are routing emails to the LLM Queue folder (not a subfolder with a different name). The folder name must match exactly what's in ~/.phd_os_config.json (default: LLM Queue).

Email pipeline: Ollama errors

Make sure Ollama is running: ollama serve in a terminal, or check that the Ollama menu bar app is active. Then verify the model exists: ollama list should show phd-email-parser. If not, run bash scripts/model/build_model.sh again.

Email pipeline: AppleScript permission errors

macOS requires explicit permission for scripts to control apps. Go to System Settings → Privacy & Security → Automation and make sure Terminal (or whichever app you use to run the pipeline) has permission to control Mail. You may also see a one-time permission dialog the first time the script runs — click Allow.

Review UI: project dropdown is empty

The dropdown fetches your project list from Todoist on page load. If it's empty, your Todoist token may be missing or expired. Check ~/.phd_os_config.json has a valid todoist_token. You can test it manually: curl -s https://api.todoist.com/api/v1/projects -H 'Authorization: Bearer YOUR_TOKEN'. If that returns projects, restart the review server.

Review UI: "Show email" section is missing

The email snippet is stored in the queue at parse time. Tasks queued before the snippet feature was added won't have one — approve or dismiss those tasks and the next pipeline run will populate snippets for new emails.

Review UI: PDF attachment link returns "Not found"

Attachments are copied from the temp directory to ~/.phd_os_attachments/ during the pipeline run. Check that directory exists and contains the file. If it's empty, the pipeline either ran before the attachment feature was added, or the PDF copy failed (check /tmp/phd_os_pipeline.log for "WARNING: Could not save attachment").

Task appears in Todoist Inbox instead of a project

A task with no project ID goes to Inbox. This happens if you approved the task with "— project —" still selected in the dropdown, or if the project list failed to load. You can move the task in Todoist manually. For future tasks, make sure the project dropdown shows your projects before approving.
